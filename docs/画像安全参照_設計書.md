## 0. 本当の目的（最優先・実装中に見失わないこと）

**翻訳のために、メルカリへ自動アクセスせず、安全に画像を参照できるようにする。**
これが唯一の目的。重さ・枚数・実装の都合は副次。実装を進めると目的を見失いやすいので、各ステップでこの一文に立ち返ること。

## 1. 現状の問題（Fact・確認済み）

- 翻訳（一括シート route② / とりこみ君のAI翻訳）が、商品画像の**メルカリURL（static.mercdn.net）をそのままAI（OpenAI等）に渡し**、AIのサーバーがメルカリへ自動アクセスしている。
  - 一括シート: `EbayTranslationApi.gs` の `buildOpenAIRequest_` / `buildGeminiRequest_` が `image_url = メルカリURL`。Gemini経路は `fetchImagesBatch_` がGAS側でメルカリをfetch。
  - とりこみ君: `background.js` `handleAiTranslate`（~265行）が `input_image, image_url = メルカリURL` をOpenAIへ。
- これは「メルカリへの自動的な直接アクセス」。狭義のページscrapingではないが、避けたい挙動（eBayアカウント保護の観点でも）。

---

## 2. 確定した方針（椛島さん決定 2026-05-21）

- **安全な参照方法** = 画像をシートに**セル内画像（CellImage）**として持ち、翻訳は **`getContentUrl()`（Googleホストの公開URL）** を読む＝メルカリ非経由。
- **非メルカリ化はトップ画像（商品画像1）の1枚だけ**。商品画像2〜20は**従来通り `=IMAGE(メルカリURL)`**（目視チェック用）。
- **翻訳**:
  - トップ画像 → セル内画像の getContentUrl を参照（安全・メルカリ非経由）。
  - **ユーザーが2枚以上を選んだ時だけ**、2枚目以降は `=IMAGE` のメルカリURLから取得（その分はメルカリアクセスが発生＝ユーザー判断で許容）。
- アカウントは**個人**（GAS 1日90分枠）。1枚＝約1.4秒/件で重さは実質ゼロ。
- **改修順序: ①とりこみ君 → ②一括シート**。

## 3. 実証済みの事実（この設計の裏付け・すべて実機確認済み）

1. **ブラウザでのbase64取得（とりこみ君側）**: MV3で `host_permissions` に `https://static.mercdn.net/*` を追加し、**background service worker** で `fetch → arrayBuffer → btoa（32KBチャンク）` → **成功**（status200 / 249,239 bytes / base64 332,320 / 14ms / CORSエラーなし / 画像描画OK）。
   - `canvas` は他オリジン画像で汚染（tainted）するため**使わない**。content scriptのfetchはCORS対象なので**background SWで行う**のが要点。
2. **GASでのセル内画像化（書き込み側）**: `SpreadsheetApp.newCellImage().setSourceUrl("data:image/jpeg;base64," + b64).build()` を `setValue` → CellImage化（**約1.4秒/枚**）。`getContentUrl()` が `lh*.googleusercontent.com` の**公開URL**を返す（HTTP200 / image/jpeg / 元とバイト同一 / 認証不要で外部取得可）。
3. **翻訳での読み取り**: getContentUrl の Google画像から、AIが**カード番号まで正確に読めた**（ヤドキング 015/080 XY9、ピカチュウVMAX 031/100 RRR、PSA認証番号も）。
4. ※ `getContentUrl` / `newCellImage` は **Apps Script専用**（Sheets REST APIでは不可）。`clasp run` はAPI executable未設定で不可。GAS実行は「ウェブアプリ(doGet)をユーザーがデプロイ」or「エディタで関数実行」で、いずれも**ユーザーによるscope承認（ブラウザ）が必要**。

## 5. 改修内容

### Phase 1: とりこみ君（先に実装）
1. `manifest.json`: `host_permissions` に `"https://static.mercdn.net/*"` を追加。
2. `background.js`: 画像URL→base64 取得関数を追加（**実証済みパターン**: `fetch → response.arrayBuffer() → 32KBチャンクで String.fromCharCode → btoa`）。失敗時はエラーを握ってURLのみで継続。
3. 出力フロー（`content.js` / `background.js` / `background-queue.js`）: シート出力時、**トップ画像（商品画像1）だけ base64 を取得**し、doPostへ送るデータに含める（商品画像2〜20は従来通りURLのまま）。
4. `doPost`（`options.js`）: 受け取ったデータで、**商品画像1のセル**は `base64 → newCellImage → setValue`（セル内画像）、**商品画像2〜20**は従来通り `=IMAGE(URL)`。
   - doPost変更は、ユーザーのシートのApps Scriptへ反映が必要（配布GAS更新 → ユーザー貼り替え/再デプロイ → scope承認）。
- **受け入れ条件**: メルカリ商品を取り込むと、商品画像1がセル内画像（getContentUrlが取れる）、商品画像2〜20が=IMAGE。既存の取り込み機能・他サイト取り込みは不変。

### Phase 2: 一括シート（翻訳）
1. `EbayTranslationApi.gs`: AIに渡す画像の組み立てを変更。
   - **トップ画像（商品画像1）** = セルの CellImage の `getContentUrl`（Googleホスト）を使う（**メルカリURLを使わない**）。
   - **2枚目以降**（`EBAPI_GEMINI_MAX_IMAGES_PER_ITEM` が2以上でユーザーが選んだ分）= 従来通り `=IMAGE` のメルカリURL。
   - OpenAI経路（`buildOpenAIRequest_`）・Gemini経路（`buildGeminiRequest_` / `fetchImagesBatch_`）の両方を、この方針に合わせる。
2. ルート/Library 同期。
- **受け入れ条件**: トップ画像のみの翻訳でメルカリへの取得が発生しない（AI/GASがGoogleのみを取得）。カード番号が出力に反映。2枚以上選んだ時のみ追加分がメルカリ取得。既存翻訳機能は不変。

（補足: とりこみ君自身のAI翻訳 `handleAiTranslate` を運用で使うなら、そちらも同方針＝トップ画像はbase64 data URL直渡し、2枚目以降はURL。）

## 12. 実装1回目の失敗と修正点（2026-05-21・親が診断）

1回目の実装はテスト失敗（商品画像1が =IMAGE のまま、セル内画像化されず）。原因（すべてコード確認済み）:

- **ユーザーの実操作は「直接インポート（直接エクスポートボタン `content.js:1586`）」と「内容確認→スプレッドシートに追加」**。どちらも `exportToSpreadsheet`(5266) → `action:'exportToSheet'` → `handleExportToSheet`(background.js:182) 経路。
- だが base64化（`fetchImageAsBase64`）は `background-queue.js:399` の**キュー経路（＝「両方へエクスポート」専用）にしかなく**、実操作経路を通らない → base64 未送信。
- `doPost`(Main.js:267) の**単一 `{values}` モード正規化(295行)が values/sheetName のみ拾い `topImageBase64` を捨てる**。
- `doPost`④の newCellImage ブロックは repo にあるが**4シートに未push**（ライブは旧版）→ 両方へエクスポートで送っても無視される。
- **v5ガード未実装**。

### 修正点（新セッション）
1. **base64取得を全送信経路に**：`handleExportToSheet`(background.js:182) にも base64ステップを入れる（or 直接インポート/内容確認をキュー経路 enqueueExport に統一）。送信payloadに `topImageBase64` を含める（`exportToSpreadsheet` 側で先頭画像URL `topImageUrl` を持たせる必要あり）。
2. **doPost 単一モード(295)で `topImageBase64` を引き継ぐ**（`rows = [{ values, sheetName, topImageBase64: data.topImageBase64 }]`）。
3. **doPost newCellImageブロック(486)に `sheetName==='インポート用'` ガード追加**（v5のH列＝仕入先コード破壊防止）。
4. **doPost を4シートに push＋webhook 再デプロイ（要ユーザー承認）**。

### 既実装（再利用・再作不要）
①manifest mercdn 追加 / ②`fetchImageAsBase64`(background.js:22) / ③キュー経路base64(background-queue.js:399) / ④doPost newCellImageブロック(Main.js:486、※v5ガード追加要)。

### テスト（実機・実操作で）
「直接インポート」「内容確認→スプレッドシートに追加」で取り込み → 商品画像1（I列）が**セル内画像（数式バー空）**、2〜20は=IMAGE、**v5インポートのH列が壊れていない**ことを確認。

## 13. webhook / デプロイの運用と障害切り分け（2026-05-21・実装2回目の教訓）

**実装コード（クライアント送信・doPost）は正しいのに =IMAGE が続いた。原因はコードでなく webhook/デプロイの「配線」。** 忘れると必ず再発するので必読。

### 13-1. 4シート × webhook の構造（Fact）
- ユーザーは4つのスプレッドシート（各々 bound script・固有 scriptId）を使い、とりこみ君は**1つの webhook URL（/exec）に送る**。取り込み先シートごとに webhook が異なる。
- 各シートには複数の deployment があり、webhook URL は**特定の deployment ID** を埋め込む。

### 13-2. デプロイの2種類（核心・Fact）
| 操作 | 効果 | とりこみ君の貼り直し |
|---|---|---|
| `clasp push` | **HEAD（最新コード）を更新するだけ**。固定バージョン(@N)を指す webhook は変わらない＝旧コードを配信し続ける | — |
| `clasp deploy -i <deploymentId>` | 既存 deployment を新バージョンに更新。**URL不変** | **不要** |
| 新規 deploy | 新しい URL が出る | **必須（貼り直さないと旧版のまま）** |

→ **push しただけでは webhook に反映されない。** 固定版 webhook は `clasp deploy -i` で中身を更新する。

### 13-3. 今回のデプロイ状況（Fact, 2026-05-21）
- **sheet#3**（webhook `AKfycbxAjEOwQ...`）= @6 → **@7 に再デプロイ済み（新doPost配信中）**。URL不変。
- **sheet#1 / #2 / #4** = コードは HEAD に push 済みだが、**webhook は未再デプロイ（旧doPostのまま）** → これらに取り込むと =IMAGE。ユーザーがそのシートも使うなら同様に `clasp deploy -i` で再デプロイ要。

### 13-4. 拡張機能2個の罠（Fact）
- **ローカル版（load unpacked・base64改修あり）と公開版（ストア・無改修）は storage が別**＝webhook 設定も別。**base64 を送るのはローカル版だけ**。公開版のボタンを押すと必ず =IMAGE。
- 切り分け時は**公開版を無効化**し、ローカル版だけにする。**ローカル版の webhook 設定が、再デプロイした deployment（@7）の URL と一致しているか**を必ず確認。

### 13-5. doPost のシート名ガード（Fact）
- 送信（`exportToSpreadsheet` content.js:5470）は `sheetName: selectedSheet.sheetName`（**選択したシート名**）を送る。
- doPost の newCellImage は **`sheetName === 'インポート用'` の完全一致**時のみ実行。→ 選択シート名が厳密に「インポート用」でないと =IMAGE。

### 13-6. 障害切り分けチェーン（=IMAGE が続くとき、この順）
1. **公開版を無効化**・ローカル版だけ有効化／拡張リロード。
2. **ローカル版の webhook 設定 = 新デプロイ(@7)の URL** か。
3. **取り込み先がその webhook のシートか**（別シート #1/#2/#4 なら未再デプロイ＝旧コード）。
4. **選択シート名 = 厳密に「インポート用」** か。
5. **webhook へ直接テスト送信**（base64付き・sheetName='インポート用'）→ セル内画像が入れば doPost 正常＝原因はクライアント配線。=IMAGE なら doPost側（再デプロイ未反映/例外）。

### 13-7. 解決（Fact・2026-05-21・椛島さん実機確認）
- **画像が入った（I列＝セル内画像）＝動作OK**。
- **根本原因は「デプロイ」**＝webhook が旧固定版（@6）を指していたこと。**拡張機能のコード（修正1/2/3）は最初から正しかった**＝コード/機能の問題ではない。
- 修正＝`clasp deploy -i` で sheet#3 の webhook を **@7（新コード）に再デプロイ**したこと。
- **教訓**：`clasp push` は HEAD のみ更新し、固定版を指す webhook は変わらない。**固定版 webhook の更新は必ず `clasp deploy -i`（URL不変）**。
